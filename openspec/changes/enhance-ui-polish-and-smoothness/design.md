## Context

当前应用已具备完整的 AI 项目展示能力，但首页和详情页在视觉一致性、过渡连贯性与状态反馈上仍偏功能导向，缺少“高品质产品化”的体验。现有界面在多端展示时存在局部信息密度不均、组件节奏不统一、动画语义不足等问题。此次设计目标是在不改动核心业务逻辑的前提下，对视觉系统、动效系统与反馈系统做一次系统化升级，并确保性能可控。

## Goals / Non-Goals

**Goals:**
- 建立统一的视觉 token 与样式层级，使页面更美观、更有秩序。
- 为关键交互路径补充有意义的过渡与进入动效，提升“丝滑感”。
- 统一加载、空态、错误、成功反馈组件与文案，提升可理解性。
- 优化移动端与桌面端布局策略，减少跳动、拥挤与断层感。
- 在动效增强后仍保持良好性能表现（滚动、筛选、刷新场景）。

**Non-Goals:**
- 不更改后端接口契约和数据库模型。
- 不重写页面信息架构，不改变核心业务流程。
- 不在本阶段引入大型动画框架或重量级 UI 库替换。

## Decisions

1. 采用分层视觉 token（颜色、间距、圆角、阴影、透明度、动画时长）。
   - Rationale: 统一设计语言并降低后续维护成本，避免页面“拼贴感”。
   - Alternative: 在组件内就地写样式；放弃，因一致性差且难扩展。

2. 定义动效规范为“场景级”而非“组件级泛化”。
   - Rationale: 不同场景（首屏进入、筛选切换、卡片交互）需要不同节奏和反馈强度。
   - Alternative: 统一一个过渡模板；放弃，因体验机械且缺乏语义。

3. 状态反馈组件统一抽象，覆盖 loading/empty/error/success。
   - Rationale: 避免重复实现导致文案和视觉风格割裂。
   - Alternative: 页面各自处理状态；放弃，因维护成本和一致性风险高。

4. 响应式优化遵循“内容优先 + 控制密度”的布局规则。
   - Rationale: 在小屏设备上减少拥挤和视觉噪音，提升可读与可点性。
   - Alternative: 仅微调断点；放弃，因无法系统解决密度问题。

5. 性能策略采用“轻量动画 + 渲染约束 + 可降级”方案。
   - Rationale: 在流畅体验与性能之间取得稳定平衡。
   - Alternative: 高强度动画覆盖全局；放弃，因低性能设备易卡顿。

## Risks / Trade-offs

- [动效过多导致低端设备卡顿] -> Mitigation: 限制并发动画、缩短时长、提供 reduced-motion 降级路径。
- [视觉升级后与既有组件风格不一致] -> Mitigation: 先梳理基础 token，再逐步替换高频组件。
- [状态反馈统一后文案不够贴合业务场景] -> Mitigation: 统一骨架 + 页面可注入业务文案。
- [响应式调整引发局部布局回归] -> Mitigation: 覆盖关键断点截图检查与回归测试。

## Migration Plan

1. 新增并接入视觉与动效 token，不直接破坏旧样式。
2. 优先改造首页核心区块（头部、筛选、列表、状态区），再覆盖详情页。
3. 引入统一状态组件并替换页面分散实现。
4. 完成响应式回归与性能检查（构建体积、渲染、滚动、筛选切换）。
5. 如出现明显回归，按模块回滚（token 层、动效层、页面层可独立回退）。

## Open Questions

- 是否需要提供“低动效模式”开关（除系统 reduced-motion 外）？
- 动效节奏是否按“信息密度”自适应（列表越多，动效越轻）？
- 首页和详情页是否采用同一视觉主题强度，还是首页更强、详情更稳？
